==============================================================================
 1. VERDICT
==============================================================================
The method is sound and the core code is clean. The central idea — separate
compositional change (composition blocks on the SAMPLE-REMOVED view Z_clust) from
cell-state change (RMD displacement on the SAMPLE-PRESERVED view Z_rmd) — is
principled, and the code enforces that separation where it matters (cell typing runs
on Z_clust in every path I checked). The signature failure mode (batch leaking into
the sample embedding) is actively defended.

The issues are not in the idea. They are, in order of importance:

   A. REPRODUCIBILITY / PATH-DEPENDENCE  -> the same input can give different results
      depending on CPU-vs-GPU node, and several stochastic steps are unseeded.
      This is the one that can affect numbers in the paper.

   B. SILENT FAILURE / USABILITY  -> crashed pipelines exit 0; the documented
      quick-start command is rejected by config validation; a pandas-2 crash in
      RAISIN is swallowed.

   C. MAINTAINABILITY / HYGIENE  -> 250-parameter wrapper, no .gitignore, broken
      requirements file, a memory-bomb util, doc drift.

The autotune / parameter-tuning behavior is NOT in this list. After discussion it is
confirmed to be an intentional semi-supervised feature, and the paper keeps its
benchmark comparisons fair by using the untuned default — see Section 6.

Strengths to preserve: blocks.py (clean, backend-agnostic math); the Z_clust/Z_rmd
discipline; the fairness-conscious benchmark design (untuned default + the corrections
already in figure/error_check/); the broad downstream suite; strong internal docs.

 P0 — Integrity / reproducibility. Close before generating further paper results.
 ----------------------------------------------------------------------------

 P0-1  Make CPU and GPU preprocessing numerically equivalent.            [verified]
   Where  preparation/rna_preprocess_cpu.py:249-257 does a per-sample min_cells filter
          AND a final filter_genes(min_cells=0.001*n_obs); rna_preprocess_gpu.py:225-245
          does NEITHER (and drops percent_top=None in calculate_qc_metrics).
          Same class of gap in atac_preprocess_cpu vs _gpu [audit]; and CPU cell-typing
          Leiden uses flavor="igraph", directed=False while GPU uses bare rsc.tl.leiden
          -> different communities [audit].
   Why    Which node ran the job silently changes the gene/sample set -> the cell types
          -> composition -> X_DR_sample -> every downstream number.
   Fix    Extract ONE shared QC/filter/HVG/cluster routine that both paths call; the
          only allowed difference is the compute backend. Add a parity test (P3).
          Until parity holds, pin all reported numbers to ONE path and record which.

 P0-2  Seed TSCAN origin selection.                                      [verified]
   Where  sample_trajectory/TSCAN.py:838  origin = random.choice(ends)  (`random`
          imported but never seeded). trajectory_diff_gene.py:753-754 inherits the sign.
   Why    The origin decides whether main_path is reversed -> pseudotime DIRECTION
          flips run-to-run -> every trajectory-DGE UP/DOWN label and Fig 5 sign can
          invert.
   Fix    Choose the origin deterministically (e.g. endpoint with the smaller mean of
          the trajectory label), or seed `random`. Anchor DGE direction to an external
          variable (corr with sev.level), not raw pseudotime sign.

 P0-3  Fix RAISIN: pandas-2 crash + unseeded permutation.                [verified]
   Where  sample_clustering/RAISIN_TEST.py:239,828  means_df.groupby(groups, axis=1)
          -> TypeError on pandas>=2 (axis=1 removed), caught by a try/except that only
          prints a warning, so the heatmaps silently never render.
          RAISIN_TEST.py:562  np.random.permutation(...) unseeded -> p-values/FDR not
          reproducible.  RAISIN.py:802  np.random.seed(42) mutates the GLOBAL RNG.
   Fix    means_df.T.groupby(groups).mean().T ; thread a seed and use a local
          np.random.default_rng(seed).permutation ; replace the global seed with a
          local default_rng.

 P0-4  Stop silent pipeline failure (crashed run must not exit 0).       [verified]
   Where  SampleDisc.py:61-66 outer try/except prints str(e), drops the traceback;
          wrapper.py per-pipeline try/except stash results['*_error'] and CONTINUE
          [audit ~1220,1370,1604].
   Why    A consumer (or the experiment-runner) reading exit code / summary treats a
          crashed pipeline as complete — dangerous for benchmarking.
   Fix    Re-raise after logging, or exit nonzero when any *_error is present; print
          traceback.format_exc() at top level; make "continue on error" opt-in (False).

 P0-5  Make the multiomics cluster-role key fallback a hard error.       [verified]
   Where  wrapper/multiomics_wrapper.py:31-33  the cluster (composition) role falls back
          to X_glue (the SAMPLE-PRESERVED view) when Z_clust is absent; it only raises
          if the key is wholly missing. The docstring itself calls this "unsupported".
   Why    On a partial/broken run this routes sample-preserved signal into the
          composition blocks — the exact leakage the method prevents — silently.
   Fix    The RMD-role fallback to X_glue is fine; the CLUSTER-role one is not. Require
          Z_clust for the cluster role; raise a clear error otherwise.

 6. NOTE ON THE AUTOTUNE FEATURE (design intent — confirmed with author)
==============================================================================
 Autotune is an INTENTIONAL semi-supervised feature, not a leak. The user supplies the
 real labels and the tuner refines the embedding (e.g. the RMD weight / resolution) to
 surface that biological axis. Tuning on all labeled samples — with NO train/test split
 — is the intended behavior, analogous to supervised dimensionality reduction. There is
 no claim of held-out generalization, so an in-sample objective is appropriate.

 The paper keeps its benchmark comparisons fair by reporting the UNTUNED default
 configuration (paper §2.3: "without using evaluation labels for parameter tuning, so
 that comparisons against unsupervised baselines remained fair"). The tuned result is
 not used in the head-to-head comparisons. This is the right call and resolves the
 fairness concern.

 Therefore the earlier "circular evaluation" item is WITHDRAWN. Only light, optional
 tool-quality suggestions remain (none are blockers):

   - Document the semantics in the tool/docs and in the autotune output, so a downstream
     user does not read a TUNED association p-value as an unbiased discovery test.
   - Record tuned-vs-untuned (and the tuned alpha + target) in the sample-embedding /
     association output for provenance.
   - Optional robustness: at small n the in-sample proxies can pick a degenerate alpha;
     a light sanity bound or the existing CV-kNN proxy could make the choice steadier.
     Quality-of-tuner only, not correctness.

 One-line confirm for the paper: ensure Supplementary Table 3 (§2.5, the tuned top-1
 ranking) is labeled as a supervised "with tuning you can focus further" analysis,
 distinct from the untuned fair comparison — so a reader can't mistake it for the
 benchmark result.
